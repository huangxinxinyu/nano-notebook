# Revoke access before asynchronous purge

Destructive product commands will synchronously revoke visibility and future use in PostgreSQL, cancel affected active Runs, and create durable idempotent purge work owned through the relevant Module interfaces. Minimal Deletion Tombstones may remain solely to enforce unavailability and coordinate cleanup; they expose no deleted content and are not restorable. Purge Jobs remove Qdrant projections, Blob Store objects, derived artifacts, private Chat data, and retained Trace payloads according to ownership and reference rules, and retries cannot make content visible again. Production launch must define backup expiry and Provider retention/deletion guarantees, but active-product access revocation cannot depend on those slower systems.
