# Schedule Jobs with leases and fixed Workload Classes

PostgreSQL will remain the only durable Job queue authority. Workers will claim eligible Jobs atomically with expiring leases, attempts, and heartbeats, and expired work may be reclaimed, giving at-least-once execution whose external effects and stage commits must be idempotent. Fixed interactive Agent, Source Processing, and offline Eval/Reindex Workload Classes will have separate concurrency budgets and priority so background work cannot consume all capacity; the product's one-active-Agent-Run-per-User rule is enforced when work is admitted. `LISTEN/NOTIFY` may reduce wake-up latency but polling remains the recovery path and notifications never become queue truth. Values not fixed by an owning Sprint remain detailed-design parameters for their Job kind.

For Sprint 2B Agent Jobs, the Job row stores only the current execution generation: an incrementing attempt number, a random Lease Token, and its expiry. Claim or reclaim replaces that token atomically; heartbeats extend the lease conditionally using PostgreSQL time; and every failure or publication write must prove that the same token remains current and unexpired. This is fencing for recovery, not an exactly-once claim. A separate durable Attempt history is deferred to the Sprint 4 Durable Agent Trace.

Agent Jobs allow three total execution attempts. Lease expiry may start attempts two and three, but expiry of attempt three fails the active Run and Job with `recovery_exhausted`; explicit terminal model failures never consume another Job attempt. This is a bounded recovery safety valve, not a claim that three attempts make execution exactly once or guarantee success.

Sprint 2B uses a configurable default 30-second Agent Job lease renewed every 10 seconds. Renewal conditionally replaces the deadline with PostgreSQL current time plus 30 seconds; it never extends from the old deadline. Failure to prove the current unexpired Lease Token prevents renewal and publication.

Expired running Agent Jobs are reclaimed directly in the same `FOR UPDATE SKIP LOCKED` claim transaction and never pass back through `queued`. The reclaim keeps the Job and product Run `running` while atomically rotating the execution generation; expiry at the three-attempt limit terminalizes the Job and Run instead.
