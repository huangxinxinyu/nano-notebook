# Separate Operational Telemetry from Durable Agent Traces

OpenTelemetry-compatible Operational Telemetry will carry correlated traces, metrics, and logs across the Control Plane, Jobs, Workers, Bifrost, PostgreSQL, Qdrant, and Blob Store for diagnosis and performance analysis, and may be sampled or expired by operational policy. The Agent Module will separately retain the mandatory Durable Agent Trace so Agent execution can be reconstructed independently of telemetry availability. Common `trace_id`, `run_id`, `job_id`, and `attempt_id` values will connect the two systems, while SSE remains only a client projection. The Durable Agent Trace schema, payload semantics, storage split, access, sampling prohibition or exceptions, and retention remain explicitly deferred to detailed design.

Sprint 3 removes the single-call `iteration_count`, `finish_reason`, and token-usage columns from Agent Run rather than redefining them as misleading multi-step aggregates. Decision count is derived from Run Checkpoints for recovery, while per-Model-Call finish reason, usage, latency, Provider retries, and complete cost accounting belong to the Sprint 4 Durable Agent Trace and Model Call records.
