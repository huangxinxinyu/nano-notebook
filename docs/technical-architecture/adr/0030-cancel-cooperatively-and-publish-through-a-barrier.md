# Cancel cooperatively and publish through a barrier

Stop, Source-deletion, membership-removal, and Notebook-deletion commands will first persist cancellation or invalidation in authoritative PostgreSQL state. Workers will reauthorize and observe that state at bounded execution boundaries, request cancellation of in-flight dependencies where supported, and reject late results through fencing and state-version checks. A draft becomes a durable Assistant Message and Citations only through one PostgreSQL Publication Barrier that revalidates the Principal, Notebook, complete Run Evidence Set, Source availability, and Citation resolution in the same transaction. Partial streamed text and work completed after cancellation remain non-authoritative and cannot become a completed Grounded Answer.

For a user Stop command, a committed cancellation makes the Agent Run immediately and terminally `cancelled`, releases the active-Run slot, and guarantees that no Assistant Message can later be published. Cancelling an in-flight Provider request is cooperative and best-effort rather than part of that product guarantee. Publication and cancellation serialize on authoritative Run state: whichever transaction commits first wins, and a completed answer is not retroactively withdrawn. Recovery observes a cancelled Run and cleans up its delivery records without re-executing the model call.

The Stop transaction also marks its Agent Job `cancelled`. Sprint 2B uses the existing 10-second Worker heartbeat as the cancellation propagation boundary instead of adding another notification listener: the next conditional renewal affects zero rows and cancels the local execution context. This delay may waste bounded Provider work but cannot publish a late answer, because the Publication Barrier independently requires a running Run and Job plus the current unexpired Lease Token in its commit transaction.

Loss of a Publication commit acknowledgement is treated as an unknown infrastructure outcome, not as a failed Run. The Worker reloads the authoritative Run and Job before deciding: it accepts an already completed publication, may retry the same in-memory result while its lease remains current, and discards the result after cancellation or lease loss. If it cannot reconcile with PostgreSQL, it stops renewing and leaves recovery to distinguish committed completion from a still-running Job. Publication uniqueness and lease fencing make that reconciliation safe.
